## task-004 — Phase 3: SSH tunnel lifecycle manager (DONE 2026-06-10)
- Extended src-tauri/src/tunnel.rs (files array = ONLY tunnel.rs; no lib.rs change needed —
  tunnel is already `pub mod`'d). Kept it std-ONLY on purpose so the standalone
  `rustc --test --edition 2021 src/tunnel.rs -o /tmp/t && /tmp/t` validation still works:
  51 tests green (30 from task-003 + 21 new). Full `cargo test` still blocked by the
  libdbus/webkit sys-dep wall (unchanged from task-003).
- DESIGN: TunnelManager { forwards: Vec<ForwardProcess> }, Send+Sync (all-std fields).
  Exposed `pub type SharedTunnelManager = Arc<Mutex<TunnelManager>>` as the intended
  Tauri managed-state shape (task-005 wires it). ONE ssh child per forward (not a single
  multi -L) so each forward has independent status + teardown.
- TEARDOWN (key decision): RAII via `impl Drop for ForwardProcess` (kill + wait-to-reap +
  join stderr thread). TunnelManager::stop() just does `self.forwards.clear()` — dropping
  each ForwardProcess reaps it. This is the leak guarantee on stop/exit/panic-unwind; only
  app SIGKILL can still leak (documented). task-005 must call stop() from tauri
  RunEvent::Exit/ExitRequested; Drop is the backstop. Note: kill() on an already-exited
  child errors on some platforms — ignore it (`let _ =`).
- STATUS INFERENCE: pure `classify_ssh_log_line(&str) -> Option<SshLogSignal>` (lowercases,
  checks FAILURES BEFORE success so a late fatal line wins) + a background thread per child
  reading ChildStderr line-by-line folding signals into Arc<Mutex<ForwardStatus>>. Reconciled
  in ForwardProcess::current_status() strongest-first: child.try_wait() (exited→Stopped if
  success else Failed) > active probe_local_port() (open loopback = Connected) > thread status.
  The 0600 gotcha shows up BOTH as a pre-flight SshKeyError::BadPermissions (with path +
  chmod 600 hint) AND as a runtime stderr SshFailure::KeyPermissions ("bad owner or
  permissions" / "unprotected private key file").
- KEY RESOLUTION: ssh_key_candidates(project_id, home) models sandbox.js resolveProjectSshKey
  precedence (<id>.key -> global.key -> ~/.ssh/id_rsa). resolve_ssh_key_with(cands, exists_fn)
  takes an injected predicate so precedence is unit-tested WITHOUT touching the FS; the real
  resolve_ssh_key uses `Path::is_file`. Only the key PATH is passed to ssh via -i — contents
  never read (mode checked via PermissionsExt under #[cfg(unix)]; #[cfg(not(unix))] no-op).
- GOTCHA (probe test without a live server): bind TcpListener to 127.0.0.1:0 to learn a free
  port, DROP it, then probe_local_port(port) must be false. Deterministic, no SSH/server.
- ssh args (pure build_ssh_args): -i <key> -N -v -o BatchMode=yes (no GUI pw-hang) -o
  ExitOnForwardFailure=yes (so a failed forward EXITS → Failed detection works) -o
  ServerAliveInterval=15 -L <local>:localhost:<remote> <user>@<host> (target LAST).
- Used `reader.lines().map_while(Result::ok)` (needs Rust 1.57+; cargo 1.96 fine) instead of
  the deprecated-ish `.flatten()` on a Lines iterator.

## task-006 — Phase 5: Dashboard webview UI (setup form + live status) (DONE 2026-06-10)
- Created src/index.html + src/main.ts (the two in-scope files) AND extended scope to
  tsconfig.json, package.json, src-tauri/tauri.conf.json, .gitignore — each REQUIRED to
  make the frontend actually load/typecheck (recorded via `jonggrang task update --files`,
  same precedent as task-002..005). Rationale below.
- NO-BUNDLER REALITY (key gotcha): the project is a static frontend (tauri.conf
  frontendDist="../src", no Vite/dev server). A browser/webview therefore CANNOT resolve
  bare ESM specifiers like `@tauri-apps/api/core` at runtime, and node_modules is NOT part
  of frontendDist so an import map can't point at it either. SOLUTION: set
  app.withGlobalTauri=true in tauri.conf.json → Tauri injects `window.__TAURI__` mirroring
  the api package (window.__TAURI__.core.invoke / .event.listen). main.ts calls through
  that global. To stay typed without a runtime import, use TYPE-ONLY queries:
  `type InvokeFn = typeof import("@tauri-apps/api/core").invoke;` (and event.listen) — tsc
  erases these, so the emitted main.js has ZERO bare imports (verified via grep). This is
  the literal `@tauri-apps/api` invoke()/listen() the AC asked for, just reached globally.
- EMIT GOTCHA: task-001 set tsconfig noEmit:true as a "typecheck gate". But with a static
  frontendDist and beforeBuildCommand="tsc", the served dir needs main.js — noEmit meant it
  was NEVER produced (build would ship index.html with no script). Flipped noEmit:false so
  `tsc` typechecks AND emits src/main.js in place (no outDir; include:["src"]). Added
  `src/*.js` to .gitignore — it's a build artifact, regenerated by the tsc gate, do NOT
  commit it (only index.html + main.ts are committed).
- TOOLCHAIN GAP fixed: package.json devDeps had @tauri-apps/cli but NOT `typescript`, so the
  `tsc` gate had no compiler (npx pulled the bogus tsc@2.0.4 stub). Added typescript ^5.
  VALIDATION: `npm install` works here (network OK), then `./node_modules/.bin/tsc` exits 0
  (typecheck green + emit). No eslint/test framework configured (AGENTS.md: test cmd is a
  no-op echo), so lint/test steps are N/A for the frontend.
- isolatedModules: main.ts uses only `import type` (erased) so it would become a non-module
  and trip isolatedModules — added a trailing `export {};` to keep it a module (also needed
  for the `declare global { interface Window { __TAURI__ } }` augmentation).
- COMMAND ARG SHAPE: start_tunnel's Rust param is a single struct named `args`, so JS calls
  invoke("start_tunnel", { args: {...} }) — the OUTER key is the param name. Inside, the
  StartTunnelArgs fields are camelCase (target/forwards/projectId/keyPath) per its
  serde(rename_all). Empty keyPath/projectId are OMITTED so the backend uses its key
  precedence. Forward spec is built as `-c <cid>:<port>,...` (dashboard NOT included — the
  backend always adds it at 7777). stop/status/list take no args.
- BACKEND LIMITATION surfaced honestly: the AC wants "per-tunnel and global Start/Stop", but
  task-005 only exposes GLOBAL start_tunnel/stop_tunnel (no per-forward stop command). Since
  the frontend must not add backend logic, per-forward cards expose Open/Copy actions only;
  Start/Stop are global. Noted in a UI hint + here. A future task could add a stop_forward
  command if per-forward control is wanted.
- SECRETS: only the key PATH is ever handled (typed into the form or chosen via the optional
  dialog plugin if window.__TAURI__.dialog exists); contents are never read. Config (target,
  key PATH, projectId, forwards) is persisted in localStorage under
  "jonggrang.tunnel.config.v1" so first-launch setup is skipped on later launches.
- The 0600/owner SSH-key gotcha: backend returns it as a plain Err string from start_tunnel;
  the catch shows it verbatim in the #error-banner. No special-casing needed in the UI.

## task-007 — Phase 6: GitHub Actions build workflow (mac + linux) (DONE 2026-06-10)
- Created .github/workflows/build.yml (the only in-scope file). First .github/ content in
  the repo. Triggers: push on feat/jonggrang-tunnel-desktop + tags 'v*', pull_request on the
  same branch. Single `build` job, strategy.matrix.platform = [macos-latest, ubuntu-latest],
  fail-fast:false so one platform's failure still yields the other's binary.
- WHY THE MATRIX (documented in a header comment per AC-7): Tauri/wry cannot cross-compile a
  macOS .app/.dmg from Linux (needs Apple SDK + codesign on a macOS host). So the ONLY way to
  get both binaries is native runners — macos-latest for .dmg/.app, ubuntu-latest for
  .deb/.AppImage. This workflow is the canonical source of both platform binaries.
- LINUX DEPS step is gated `if: matrix.platform == 'ubuntu-latest'` and runs FIRST (before any
  cargo step) — apt-get install libwebkit2gtk-4.1-dev libsoup-3.0-dev build-essential
  librsvg2-dev. This is exactly the sys-dep wall every prior task hit locally (libdbus→webkit);
  CI is where it actually gets satisfied so `cargo test`/`tauri build` finally compile tauri/wry.
- cargo test runs on BOTH legs via `working-directory: src-tauri` (Cargo.toml lives there).
  npm ci runs before it (tauri-action's beforeBuildCommand=tsc needs node_modules).
- PINNING (AC-7): all third-party actions pinned to specific patch version TAGS, not floating
  majors — checkout@v4.2.2, setup-node@v4.1.0, Swatinem/rust-cache@v2.7.5,
  tauri-action@v0.5.20, upload-artifact@v4.4.3. EXCEPTION: dtolnay/rust-toolchain is
  channel-based (@stable) — the Tauri-recommended toolchain action has no semver release tags;
  noted in a comment that a commit SHA is the stricter pin. Caching: npm via setup-node
  `cache: npm`; cargo via Swatinem/rust-cache with `workspaces: src-tauri`.
- RELEASE-ON-TAG (AC-6): tauri-action itself cuts the GitHub Release. tagName is a conditional
  expression `${{ startsWith(github.ref, 'refs/tags/v') && github.ref_name || '' }}` — empty
  string on branch/PR runs makes tauri-action build WITHOUT releasing; a real tag name makes it
  create/attach the Release. Needs `permissions: contents: write` on the job + GITHUB_TOKEN env.
- ARTIFACTS-ALWAYS (AC-6): a separate actions/upload-artifact step runs on EVERY trigger (branch/
  PR/tag), globbing src-tauri/target/release/bundle/{dmg/*.dmg,macos/*.app,deb/*.deb,
  appimage/*.AppImage}. name = bundles-${{ matrix.platform }} so the two legs don't collide
  (upload-artifact@v4 errors on duplicate artifact names). if-no-files-found:warn.
- VALIDATION: no YAML linter in the toolchain by default (no pyyaml); validated by parsing with
  `npx js-yaml` (VALID) + a node regex sweep asserting all 7 AC sub-criteria present. Could NOT
  run the workflow itself (no GH runner / act locally), so correctness of action SHAs/tauri-action
  behavior is by-spec, not executed — first real run on push is the live check.
